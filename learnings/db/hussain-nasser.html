ACID in Databases

The ACID properties are fundamental principles that ensure reliable processing of database transactions. They stand for Atomicity, Consistency, Isolation, and Durability.

1. Atomicity

  • Definition: Atomicity ensures that all operations within a transaction are completed successfully; otherwise, the transaction is aborted.
  • Key Points:
    • All queries in a transaction must succeed for the transaction to be committed.
    • If any query fails, all previous successful queries within the transaction are rolled back.
    • If the database goes down before a transaction commits, all changes made by the transaction are rolled back upon recovery.
    • This ensures that a transaction is an indivisible unit of work.

2. Consistency

  • Definition: Consistency ensures that a transaction brings the database from one valid state to another, maintaining database invariants.
  • Key Points:
    • Ensures data integrity by enforcing rules like referential integrity (e.g., foreign keys).
    • Consistency is defined by the user and includes application-specific constraints.
    • A transaction must leave the database in a consistent state, adhering to all defined rules and constraints.
    • Relies on atomicity and isolation to maintain consistency.

3. Isolation

  • Definition: Isolation ensures that the intermediate state of a transaction is invisible to other transactions until the transaction is complete.
  • Key Points:
    • Determines whether an inflight transaction can see changes made by other concurrent transactions.
    • Read Phenomena:
      • Dirty Reads: Reading uncommitted changes from another transaction.
      • Non-repeatable Reads: A transaction reads the same row twice and gets different values because another transaction has modified the row and committed in between.
      • Phantom Reads: A transaction re-executes a query and finds additional rows that were inserted by another transaction after the initial read.
      • Lost Updates: Occur when two transactions read the same data and update it simultaneously, resulting in one update being overwritten by the other.
    • Isolation Levels:
      • Read Uncommitted: Lowest isolation level, allows dirty reads.
      • Read Committed: Prevents dirty reads, but non-repeatable reads and phantom reads can occur.
      • Repeatable Read: Prevents dirty reads and non-repeatable reads, but phantom reads can occur.
      • Serializable: Highest isolation level, prevents all three phenomena.

4. Durability

  • Definition: Durability ensures that once a transaction is committed, it will remain so, even in the event of a system failure.
  • Key Points:
    • Ensures that committed changes are permanently recorded in the database.
    • Durability Techniques:
      • Write-Ahead Logging (WAL): Changes are first written to a log before being applied to the database.
      • Asynchronous/Synchronous Snapshots: Periodic snapshots of the database state (e.g., Redis uses asynchronous snapshots).
      • Append-Only File (AOF): Logs every operation that modifies the database, ensuring data can be reconstructed.

Together, these properties ensure that transactions are processed reliably and predictably, maintaining the integrity, reliability, and consistency of the database even in the face of concurrent access and system failures.
